设计原则 /profile /review 复用提醒 医疗关键词库

📐设计原则（先读这条）

  • 只读不改：Agent 永远只输出「问题 + 翻译 + 改法建议」文本，绝不返回会自动落盘的 diff。安全承诺是产品叙事的一部分。
  • 半干活：要给「怎么改」的具体文字方案（够显得能干活），但不替用户改。
  • 说人话：所有面向用户的 问题 字段必须用用户那一行的话，技术细节放进单独字段，用户可展开看。
  • 画像驱动：/review 的入参一定带 profile 上下文，没有画像就不该 review——这是「越用越懂你」的技术体现。

①POST/profile · 扫盘结果 → 行业画像

入参是扫盘引擎（队友本地遍历产物）的结果，出参是可编辑的行业画像。前端把出参渲染成「画像卡片」，用户可手改，改完回写。

▼ 入参 Request
{
  "scan": {
    "root": "/Users/x/projects/clinic-system",
    "fileCount": 184,
    "hits": [
      { "term": "patient",    "count": 47, "where": ["models/patient.ts", "..."] },
      { "term": "就诊",        "count": 23, "where": ["services/visit.ts"] },
      { "term": "禁忌症",      "count": 6,  "where": ["rules/medication.ts"] },
      { "term": "挂号",        "count": 31, "where": ["api/register.ts"] }
    ],
    "langs": ["TypeScript", "Vue"],
    "sampleSnippets": ["function validatePatientId(id) {...}", "..."]
  }
}
▼ 出参 Response
{
  "profile": {
    "industry": "医疗",
    "confidence": 0.92,
    "evidence": ["patient×47", "就诊×23", "禁忌症×6", "挂号×31"],
    "subDomain": "社区诊所 / 门诊管理",
    "roleGuess": "懂临床流程的开发者 / 诊所数字化负责人",
    "editable": true,
    "redlines": [
      "患者隐私字段不得明文存储",
      "就诊/挂号时段需防并发重复",
      "用药需校验禁忌症"
    ]
  }
}
字段说明
industry反推出的行业，用于驱动 /review 的翻译口径
confidence0~1，低于阈值时前端提示用户手动确认
evidence炸场用：命中的证据词，前端直接展示「我凭这些猜你是医疗的」
redlines行业红线清单，/review 据此判定红线等级
editable画像卡可手改；改后的画像作为后续 /review 的上下文

②POST/review · 画像 + 文件 → 翻译问题 + 打分 + 改法

入参带 profile 上下文 + 单个文件/片段，出参是「行业语言翻译后的问题列表」。这是报告页的数据源，也是 demo 第一幕第 4、5 步的核心。

▼ 入参 Request
{
  "profile": { "industry": "医疗", "redlines": ["患者隐私字段不得明文存储", "..."] },
  "file": {
    "path": "services/visit.ts",
    "content": "localStorage.setItem('patient', JSON.stringify(p)); ..."
  }
}
▼ 出参 Response
{
  "score": 62,
  "issues": [
    {
      "problem": "患者隐私字段裸奔了，违反医疗数据红线",   // 行业话，给用户看
      "techDetail": "patient 对象明文 JSON 存进 localStorage", // 技术细节，可展开
      "redlineLevel": "high",                              // high | mid | low
      "fix": "敏感字段不落本地存储；若必须缓存，先脱敏+加密，患者 ID 用不可逆哈希",
      "loc": { "file": "services/visit.ts", "line": 12 }
    },
    {
      "problem": "同一就诊时段会被重复挂号，候诊顺序会乱",
      "techDetail": "时段并发写未加锁/未做唯一约束",
      "redlineLevel": "mid",
      "fix": "挂号写入加乐观锁或时段唯一索引，冲突时返回「该时段已满」",
      "loc": { "file": "api/register.ts", "line": 40 }
    }
  ]
}
字段说明
score本文件/项目代码质量打分（0~100），对应「代码分析与打分」诉求
problem必须是行业话，前端主展示。这是产品差异点，别写成技术 bug 描述
techDetail技术细节，前端折叠，用户想看再展开
redlineLevel红线等级，前端用颜色区分（high 红 / mid 橙 / low 灰）
fix改法建议文本——只给建议，不自动改（半干活 + 只读不改）

③复用提醒（demo 第二幕 · 医疗最小切片）

不新增独立 API，复用 /profile 的行业判定即可。当扫描项目 B 命中与历史相同的行业、且代码里出现与行业能力库已有轮子相似的意图时，触发提醒。

▼ 提醒数据（前端弹窗用）
{
  "reuseHint": {
    "matchedIndustry": "医疗",
    "message": "你做过类似的医疗项目，行业库里有现成轮子",
    "candidates": [
      {
        "name": "患者 ID 校验",
        "fromProject": "clinic-system",
        "snippet": "function validatePatientId(id) { /* 校验 + 脱敏 */ }"
      },
      {
        "name": "就诊状态机",
        "fromProject": "clinic-system",
        "snippet": "const visitFSM = { 待就诊→就诊中→已完成→复诊 }"
      }
    ]
  }
}
边界：行业能力库就是一个 JSON + 代码片段 文件，轮子由项目 A 真实扫出后人工/半自动入库 1~2 个即可。不做通用自动抽象——那是生产级工程，48h 做不出真的。

给扫盘引擎同学：命中权重按「类别覆盖广度 ＞ 单词高频」算——命中 3 个不同类别比单个词出现 50 次更能说明是医疗。证据词回传时带出现文件，前端才能点开溯源，炸场更有说服力。
